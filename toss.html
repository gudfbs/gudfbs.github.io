이력서
QA로서 본인이 기여하고 집중한 부분과 성과(또는 실패 이후의 개선 사례)를 알고 싶어요.
제가 QA를 수행할 때 가장 중요하게 생각하는 부분은 단순 기능 검증이 아닌 실제 사용자 흐름 관점에서 서비스 안정성을 확인하는 것입니다.

저는 QA 업무뿐 아니라 서비스 운영, 기술 문의 대응, 사용자 교육을 함께 경험하며 고객이 실제로 어떤 흐름에서 서비스를 사용하고, 어떤 지점에서 혼란을 겪는지 직접 확인해왔습니다.

이 경험을 바탕으로 기능이 정상 동작하는지만 확인하는 것이 아니라, 실사용자 입장에서 흐름이 자연스러운지, 예외 상황에서 문제가 발생하지 않는지까지 함께 검증하는 방식으로 QA를 수행해왔습니다.

이 관점은 신규 웹 솔루션 QA 리딩 과정에서도 적용되었습니다. 사용자 시나리오 기반으로 전수 QA 범위를 구성하고, 실제 운영 환경에서 발생할 수 있는 사용 패턴과 예외 상황을 추가했습니다.

기획/개발 과정에서 인지하지 못한 사항을 폭 넓은 시야로 오류 항목을 조기에 찾아 오류 케이스를 늘렸고, 주요 기능 오류율을 약 54% 감소(22% → 10%)시킬 수 있었습니다.

이 경험을 통해 QA는 단순히 오류를 찾는 역할이 아니라 사용자 경험과 서비스 흐름 전체를 이해하며 리스크를 줄이는 역할이라고 생각하게 되었습니다.
빠른 개발/배포 사이클 속에서 QA를 수행해보신 경험이 있다면 작성해 주세요.
현재 운영 중인 쇼핑몰 통합관리 솔루션은 오픈마켓 정책 변경과 신규 기능 배포가 자주 발생하는 환경이기 때문에 빠른 개발 및 배포 사이클 속에서 QA를 수행해 오고 있습니다.

모든 기능을 동일한 수준으로 검증하기 어려운 상황에서는 실제 사용자 영향도가 높은 영역을 우선적으로 확인하는 방식으로 QA를 진행했습니다.

서비스 운영에 직접 영향을 주는 기능을 우선 검증 대상으로 분류하고 사용자 시나리오 기반으로 핵심 흐름을 빠르게 확인하는 데 집중했습니다.

또한 기존 운영 이슈와 반복 발생 장애 이력을 참고해 회귀 테스트 범위를 조정하며 제한된 일정 안에서도 안정적인 QA를 수행하고 있습니다.

이러한 결과로 기존 QA담당자 대비 약 24%(395건->489건) 증가된 QA 건을 처리하며 빠른 사이클에서도 안정적으로 서비스가 운영되고 있습니다.
다양한 기술 스택을 활용 했다면, 해당 기술을 실제 QA에 어떻게 활용했는지 작성해 주세요.
업무에서는 JIRA와 Confluence를 활용해 테스트 결과와 이슈 재현 절차를 문서화하고 있으며, 운영 이슈 발생 시 영향 범위와 재현 과정을 정리해 개발 및 기획과 빠르게 협업할 수 있도록 노력했습니다.

또한 VSCode 환경을 활용해 Playwright 기반 테스트 자동화를 개인적으로 학습하고 있습니다. 실무에 직접 적용한 경험은 아직 없지만, 반복적으로 수행되는 웹 시나리오 테스트를 자동화하는 과정을 직접 구성해보며 테스트 효율화 관점에서 접근하고 있습니다.

특히 웹 솔루션 QA 과정에서 특정 기능 수정 이후 직접적인 연관이 없는 다른 화면 기능에서 예상하지 못한 문제가 발생했던 경험이 있었습니다.

이 경험을 통해 실제 서비스 환경에서는 모든 기능을 사람이 매번 동일한 수준으로 검증하는 데 한계가 있다고 느끼게 되었고, 반복적으로 확인해야 하는 영역은 자동화를 통해 관리할 필요가 있다고 생각하게 되었습니다.

이후에는 QA가 변경 영향도가 높은 핵심 기능 검증에 더 집중될 수 있도록 자동화 방향을 지속적으로 고민하고 있으며, 더 빠르고 안정적인 QA 프로세스를 만들기 위해 Playwright 기반 테스트 자동화를 학습하고 있습니다.
